# 宠物综合服务平台开发设计文档

> 本文档根据 `docs/宠物商城项目需求.docx` 整理，并结合当前仓库 `backend/`、`frontend/`、`docs/` 的目录结构进行技术设计。目标是为后续编码、排期、建表、接口拆分和前后端协作提供统一依据。

## 1. 项目概述

### 1.1 产品定位

宠物综合服务平台面向宠物主人、准宠物主人、宠物商家、救助机构、宠物服务人员和平台运营人员，提供宠物商城、宠物档案、宠物社区、领养申请、活体宠物交易、宠物币积分、宠物服务预约和后台审核能力。

项目早期不建议只做单一商城，而应围绕以下闭环建设：

```text
宠物档案 -> 个性化推荐 -> 商城购买 -> 宠物币激励 -> 社区分享 -> 内容互动 -> 服务预约/复购 -> 长期留存
```

另一条核心闭环：

```text
准宠物主人 -> 浏览买宠/领养 -> 咨询/申请 -> 成功接宠 -> 创建宠物档案 -> 购买用品 -> 分享成长 -> 持续消费
```

### 1.2 V1.0 建设目标

V1.0 以“能注册、能建档、能逛商城、能下单、能发社区、能领养申请、能使用智能体、能上传私人知识库、能后台审核”为目标，先打通高频、低风险、可运营的基础链路。

V1.0 优先上线：

- 用户注册登录、用户资料、地址管理。
- 宠物档案、成长记录、基础提醒。
- 宠物商城、商品分类、购物车、订单、支付结果。
- 宠物社区、动态发布、点赞、评论、收藏、举报。
- 领养信息浏览、领养申请、审核流。
- 宠物币积分、签到、任务、流水。
- 智能体模块：AI 宠物用品导购 Agent、养宠知识问答 Agent。
- 用户及宠物详细资料模块，作为智能体和知识库生成的基础数据源。
- 私人知识库上传与 RAG 检索增强，支持外部文件上传，也支持基于用户及宠物资料生成文件后入库。
- 商家后台基础能力。
- 平台管理后台、商品审核、商家审核。

V1.0 暂缓完整开放：

- 活体宠物担保交易。
- 宠物服务预约。
- 高级会员体系。
- 宠物医疗深度服务。

## 2. 技术架构

### 2.1 架构参考

当前项目采用前后端分离加模块化后端的工程形态：

- 前端：`uni-app`、`Vue 3`，兼容 H5、Android，后续扩展小程序。
- 后端：`FastAPI`、`Uvicorn`、异步接口。
- 数据库：`MySQL 8` 存储用户、商品、订单、社区、领养、商家等业务数据。
- ORM：`SQLAlchemy 2.x Async ORM`。
- 迁移：`Alembic`。
- 缓存与验证码：`Redis`。
- 消息队列：V1.0 直接使用 `RabbitMQ`，通过 `aio-pika` 投递和消费异步任务。
- 智能体：V1.0 接入大模型工作流，优先实现 AI 宠物用品导购 Agent 和养宠知识问答 Agent。
- RAG：支持用户上传私人养宠资料、商品资料、宠物健康记录文档，使用向量库进行语义检索增强。
- 对象存储：开发期可在 `backend/generated/` 保存上传图片、视频封面、协议文件等本地资源，该目录不提交 Git；生产环境迁移到 OSS/COS/S3 + CDN。
- 支付：V1.0 直接采用双模式支付，支持支付宝沙箱模式和本地 mock 支付，通过配置切换。
- 部署：已预留 `deploy/Dockerfile`、`deploy/docker-compose.yml`、`deploy/nginx.conf` 模板，后续按真实环境调整。

### 2.2 当前技术栈

本项目技术栈参考上一项目的成熟组合，并根据宠物商城、社区、订单、智能体和私人知识库场景做适配。V1.0 建议优先使用下表技术，避免在早期引入过多异构组件。

| 领域 | 本项目建议实现 |
| --- | --- |
| 前端 | uni-app、Vue 3、JavaScript、Vue 单文件组件 |
| API | Python 3.13+、FastAPI、Uvicorn、Pydantic |
| ORM | SQLAlchemy async、`async_sessionmaker` |
| 业务数据库 | MySQL 8、`aiomysql` |
| 数据库迁移 | Alembic |
| 缓存 | Redis async client，用于验证码、登录态辅助、限流、热点缓存 |
| 认证 | JWT、`token_version`、pwdlib Argon2、FastAPI `Depends()` |
| 文件上传 | FastAPI `UploadFile`、本地 `backend/generated/`，生产环境迁移 OSS/COS/S3 + CDN |
| AI 编排 | LangChain、LangGraph，用于导购 Agent 和问答 Agent 工作流 |
| 大模型 | DeepSeek `deepseek-chat`，通过环境变量切换模型和供应商 |
| 对话记忆 | PostgreSQL、LangGraph `AsyncPostgresSaver`，用于 Agent 多轮会话记忆 |
| RAG | ChromaDB、Ollama `nomic-embed-text`、PyPDF，用于私人知识库解析和语义检索 |
| 消息队列 | RabbitMQ、`aio-pika`，用于知识库解析、向量化、提醒、订单超时关闭、审核异步任务 |
| 后台异步 | RabbitMQ worker，统一处理后台任务，不使用 FastAPI `BackgroundTasks` 承担业务异步 |
| HTTP 客户端 | HTTPX，用于调用大模型、支付、短信、物流等外部服务 |
| 邮件/验证码 | fastapi-mail；短信验证码后续可替换为第三方短信服务 |
| 支付 | 双模式支付：支付宝沙箱 H5/网页支付、本地 mock 支付、RSA2 签名/验签、支付回调幂等处理 |
| 测试 | pytest、pytest-asyncio、aiosqlite、HTTP 调试文件 |
| 部署 | Dockerfile、docker-compose、Nginx 反向代理 |

当前运行依赖建议至少包括 MySQL、Redis、RabbitMQ、Ollama 和 DeepSeek API。启用完整 Agent/RAG 能力时，需要额外准备 PostgreSQL 和 ChromaDB 持久化目录。上传文件、ChromaDB 数据和 Agent 中间产物默认保存在本地目录，生产环境必须迁移到可备份、可扩容的持久化存储。

### 2.4 后端职责划分

`backend/routers/` 只负责 HTTP 入参、鉴权依赖、响应模型和状态码。

`backend/schemas/` 定义请求体、响应体和列表分页结构。

`backend/models/` 定义数据库表结构，保持字段、索引、约束清晰。

`backend/repository/` 用于封装常用查询、创建、更新、分页和事务内数据访问。当前仅为骨架，简单查询可先放在对应 `services/` 中，业务复杂后再拆出。

`backend/services/` 放置跨表业务逻辑，例如订单创建、库存扣减、宠物币发放、审核流、支付回调、消息通知、风控规则。

`backend/core/` 放置通用基础能力，例如鉴权工具、密码哈希、JWT、缓存、支付客户端、文件存储适配、风控工具、Agent 工作流、RAG 检索和向量化工具。

`backend/settings/` 统一读取环境变量，不在业务代码中硬编码密钥、数据库地址和第三方配置。

## 4. 核心业务设计

### 4.1 用户与权限

用户类型：

- 普通用户：浏览、下单、发布动态、创建宠物档案、申请领养、预约服务。
- 实名用户：发布送养信息、参与高风险交易、申请领养。
- 商家：发布商品、发布宠物、管理订单、售后、库存、服务预约。
- 救助机构：发布领养信息、审核领养申请、跟踪回访。
- 服务人员：发布服务、设置预约时间、接单、上传服务过程。
- 管理员：审核、风控、运营配置、数据报表。

鉴权方案：

- 密码使用安全哈希存储。
- 登录后返回 JWT。
- `token_version` 用于修改密码后让旧 Token 失效。
- 管理员入口与普通用户入口分离。
- 冻结、软删除用户后禁止登录。

### 4.2 宠物档案

宠物档案是平台推荐、商城复购、社区分享和健康提醒的基础。

核心能力：

- 创建多只宠物。
- 记录昵称、类型、品种、生日、到家日期、性别、绝育、疫苗、驱虫、体重、头像、相册。
- 记录成长事件：照片、视频、体重、饮食、洗澡、美容、疫苗、驱虫、生病、就医、生日等。
- 支持健康提醒：疫苗、驱虫、洗澡、美容、体检、药品、宠粮复购、猫砂复购。

### 4.3 用户及宠物详细资料

在用户上传私人知识库文件之前，系统需要先提供“用户及宠物详细资料”模块。该模块用于沉淀结构化基础信息，并支持一键生成可入库的宠物资料文件，降低用户手动整理文档的成本。

核心能力：

- 用户详细资料：养宠经验、所在城市、居住环境、预算偏好、常购品类、喂养理念、过敏/禁忌备注。
- 宠物详细资料：宠物基础档案、体型体重、年龄阶段、绝育情况、疫苗驱虫、过敏史、慢性病、饮食偏好、用品偏好、行为特点。
- 多宠物资料独立维护，可选择某一只宠物生成资料文件。
- 资料完整度检查，提示用户补充影响导购和问答质量的关键字段。
- 支持基于用户资料、宠物档案、成长记录、提醒记录生成结构化 Markdown/TXT 资料文件。
- 生成的资料文件可直接写入私人知识库，作为 RAG 的用户专属上下文。

生成资料文件建议结构：

```text
用户养宠背景
宠物基础信息
健康与护理记录
饮食与用品偏好
历史成长记录摘要
提醒与复购信息
导购/问答注意事项
```

典型流程：

```text
完善用户资料 -> 完善宠物详细资料 -> 选择宠物 -> 生成资料文件预览 -> 确认上传到私人知识库 -> 解析分块 -> 向量化入库
```

### 4.4 宠物商城

V1.0 打通标准电商链路：

```text
商品浏览 -> 商品详情 -> 加入购物车 -> 确认订单 -> 支付 -> 发货 -> 确认收货 -> 评价 -> 发放宠物币
```

商品类目建议：

- 猫咪专区、狗狗专区、小宠专区、水族专区、鸟类专区、爬宠专区。
- 宠物食品、宠物零食、宠物玩具、宠物服饰、清洁洗护、医疗保健、宠物周边。

订单状态：

- 待支付、已支付、待发货、已发货、待收货、已完成、已取消、售后中、已退款。

售后类型：

- 仅退款、退货退款、换货、补发、破损赔付、少件补发、物流投诉。

### 4.5 领养

领养模块 V1.0 可先做“信息发布 + 申请 + 审核 + 回访记录”。

流程：

```text
浏览待领养宠物 -> 查看详情 -> 提交领养申请 -> 发布方/机构初审 -> 在线沟通 -> 面谈 -> 签署协议 -> 宠物交接 -> 回访 -> 完成
```

重点风控：

- 送养人身份认证。
- 领养人身份认证。
- 领养协议。
- 回访机制。
- 防止假领养真倒卖。
- 黑名单机制。

### 4.6 活体宠物交易

活体宠物交易风险高，建议 V1.1 再完整开放。V1.0 可只保留认证商家发布和平台审核，不开放个人自由发布。

必须具备：

- 商家认证。
- 宠物健康证明、检疫证明。
- 图片/视频真实性审核。
- 预约看宠、视频看宠。
- 电子协议。
- 平台担保交易。
- 售后观察期。
- 投诉举报和客服仲裁。
- 禁止诱导线下交易。

### 4.7 社区

社区用于提高内容留存和宠物币任务完成率。

核心能力：

- 发布图文/视频动态。
- 关联宠物档案、话题、地点、商品。
- 点赞、评论、回复、收藏、关注、举报、拉黑。
- 话题运营，如 `#新手养猫#`、`#宠物领养#`、`#宠物用品测评#`。
- 举报入口，覆盖动态、评论、用户和商品软广等场景。

### 4.8 宠物币积分

宠物币必须定义为平台积分，不做区块链代币，不支持提现、转让、买卖或现金双向兑换。

获取方式：

- 签到、完善宠物档案、首次下单、消费返币、发布优质动态、评价订单、邀请好友、领养回访上传。

使用方式：

- 兑换优惠券、抵扣部分商品金额、兑换装扮、会员体验卡、抽奖、公益捐赠权益、包邮券。

风控规则：

- 有效期。
- 异常获取可冻结。
- 关键规则变更提前公告。
- 不可用于活体宠物交易定金。

### 4.9 智能体与 RAG

V1.0 将智能体作为正式核心模块建设，不再放到后续版本。智能体主要服务两个高频场景：购买前导购和日常养宠问答。

#### 4.9.1 AI 宠物用品导购 Agent

导购 Agent 面向商城转化，基于用户宠物档案、商品库、订单历史、收藏记录和私人知识库进行商品推荐。

典型能力：

- 根据宠物类型、年龄、体重、绝育状态、过敏信息和喂养偏好推荐商品。
- 根据用户问题推荐猫粮、狗粮、猫砂、玩具、清洁洗护、保健用品等。
- 对比多个商品的适用场景、价格、规格、库存、评价和注意事项。
- 支持“预算内推荐”“新手套装”“复购提醒”“换粮建议”等导购任务。
- 推荐结果必须关联平台真实商品，不允许编造不存在的商品、价格和库存。
- 对医疗、药品、处方粮等敏感品类给出风险提示，并引导咨询兽医或专业人员。

导购流程：

```text
用户提问 -> 读取宠物档案 -> 检索商品库与私人知识库 -> Agent 生成推荐理由 -> 返回商品卡片和注意事项 -> 用户加入购物车/收藏
```

#### 4.9.2 养宠知识问答 Agent

问答 Agent 面向日常养宠咨询，回答喂养、行为训练、疫苗驱虫、洗护美容、领养注意事项、新宠到家、多宠相处等问题。

典型能力：

- 基于平台知识库回答通用养宠问题。
- 基于用户私人知识库回答个性化问题，例如宠物病历、体检报告、喂养记录、商家说明书。
- 根据宠物档案生成阶段性喂养建议、成长记录文案和提醒建议。
- 对紧急症状、疑似疾病、用药问题必须提示“仅供参考，不能替代兽医诊断”，并建议及时就医。
- 对不确定问题需要说明信息不足，避免确定性误导。

问答流程：

```text
用户提问 -> 判断问题类型和风险级别 -> 检索平台知识库/私人知识库 -> 结合宠物档案生成回答 -> 输出风险提示和后续建议
```

#### 4.9.3 私人知识库与 RAG

V1.0 支持用户上传私人知识库，用于增强导购 Agent 和问答 Agent。

支持资料：

- 系统基于用户及宠物详细资料生成的结构化资料文件。
- 宠物体检报告、疫苗记录、驱虫记录、病历摘要。
- 宠物喂养记录、过敏记录、体重变化记录。
- 商品说明书、商家护理建议、领养协议、服务报告。
- TXT、PDF、Word 文档、图片 OCR 结果，V1.0 可先支持 TXT/PDF，后续扩展更多格式。

处理流程：

```text
选择来源 -> 上传外部文件或生成宠物资料文件 -> 保存文件 -> 创建知识库文档记录 -> 文本解析 -> 分块 -> 向量化 -> 写入向量库 -> Agent 检索引用
```

上传方式：

- 外部文件上传：用户上传 TXT/PDF 等资料文件。
- 生成资料上传：用户选择某只宠物，系统读取用户及宠物详细资料，生成结构化 Markdown/TXT 文件后入库。

技术建议：

- `backend/routers/knowledge_router.py` 负责上传、资料文件生成、列表、删除和重建索引。
- `backend/core/rag_service.py` 负责文本分块、向量检索、召回结果组装。
- `backend/core/agent_workflow.py` 负责导购 Agent、问答 Agent 和多轮对话状态流转。
- `backend/services/profile_document_service.py` 负责把用户及宠物详细资料渲染为可入库资料文件。
- 开发期可使用本地向量库；生产环境按数据规模切换到独立向量数据库。
- 私人知识库必须按 `user_id` 隔离检索，不能跨用户召回。
- 删除知识库文档时要同时删除原文件、解析文本和向量索引。

## 7. 支付与订单设计

### 7.1 V1.0 支付策略

V1.0 支付模块直接采用双模式设计，避免后续再重构支付链路：

- `mock`：本地 mock 支付，用于无三方依赖的开发、演示和自动化测试。
- `alipay_sandbox`：支付宝沙箱支付，用于联调真实签名、收银台跳转、异步通知、同步回跳和支付状态查询。

支付模式通过 `PETMALL_PAYMENT_MODE` 切换，生产环境不得启用 mock 作为真实收款链路。

必须支持：

- 创建支付交易单。
- mock 支付确认接口。
- 支付宝沙箱收银台链接生成。
- 支付宝 RSA2 签名和验签。
- 支付宝异步通知。
- 支付宝同步回跳。
- 支付状态主动查询。
- 支付回调幂等更新订单。
- 支付结果页按 `out_trade_no` 查询结果。

V1.0 可暂缓真实退款能力，但需要在支付交易表中预留 `refunded` 状态。

### 7.2 统一支付交易表

建议建立统一支付交易表 `payment_transaction`：

- `out_trade_no`：平台交易号。
- `business_type`：商品订单、会员、服务预约等。
- `business_id`：业务订单 ID。
- `pay_channel`：mock、alipay_sandbox。
- `payment_mode`：当前支付模式，便于排查环境配置。
- `amount`：支付金额，单位分。
- `status`：created、paying、paid、closed、failed、refunded。
- `channel_trade_no`：三方交易号。
- `raw_notify`：回调原文，必要时脱敏存储。

支付回调只修改支付单，再由业务服务幂等更新订单。

推荐接口：

```http
POST /payments/orders/{order_id}/pay
POST /payments/mock/{out_trade_no}/confirm
GET  /payments/alipay/{out_trade_no}/page-pay
POST /payments/alipay/notify
GET  /payments/alipay/return
GET  /payments/{out_trade_no}/result
POST /payments/{out_trade_no}/query
```

- `api`：FastAPI 后端。
- `mysql`：MySQL 8。
- `redis`：验证码、缓存、限流、轻量队列。
- `vector_store`：开发期可使用本地向量库目录，挂载在 `backend/generated/vector_store/`。
- `ollama`：本地向量模型服务，可选；也可以切换为云端 embedding 服务。
- `nginx`：静态资源和 API 反向代理，按上线环境接入。

当前部署模板只用于本地或测试环境，不应直接作为生产配置。生产环境需要补充 HTTPS、域名、CORS 限制、数据库账号、备份策略和密钥管理。

后续扩展：

- `rabbitmq`：异步任务队列。
- `worker`：知识库解析入库、视频转码、提醒推送、订单超时关闭。
- `minio`：本地对象存储替代。

## 14. 关键风险与解决方案

| 风险 | 说明 | 解决方案 |
| --- | --- | --- |
| 活体宠物虚假交易 | 盗图、假宠物、骗定金 | 商家认证、平台担保、视频看宠、交易留痕 |
| 领养变倒卖 | 假领养后出售 | 实名认证、领养协议、回访、黑名单 |
| 宠物币合规风险 | 被误解为虚拟货币 | 定义为积分，不提现、不转让、不炒作 |
| Agent 编造商品 | 大模型可能输出不存在的商品或错误价格 | 推荐结果必须绑定商品 ID，价格、库存和上下架状态以数据库为准 |
| RAG 数据串库 | 私人知识库可能被跨用户召回 | 向量索引和检索条件强制携带 user_id，删除文档同步删除向量 |
| AI 医疗误导 | 用户可能把问答当成诊断 | 医疗、用药、急症场景强制提示仅供参考并引导就医 |
| 社区违规内容 | 虐宠、血腥、诈骗 | 保留举报、拉黑、下架和管理员处置机制 |
| 医疗责任风险 | 用户依赖平台建议 | 明确仅供参考，引导就医 |
| 商家服务质量不稳定 | 服务差导致投诉 | 评分、保证金、平台介入 |
| 售后纠纷复杂 | 活体宠物售后争议大 | 合同、健康证明、观察期、客服仲裁 |
| 冷启动困难 | 初期内容和商家少 | 城市试点、达人种草、机构合作 |
